# Gotchas: i18n / Internationalization

## German-only policy + Claude translations routine

Developers write **German only**. Every other locale (`en`, `fr`, `es`, `it`, `nl`, `el`, `la`, `no`, `zh`) is filled in by the [Claude translations routine](../../.claude/routines/translate.md):

1. **Combined trigger** — daily cron at 06:00 UTC **plus** a GitHub event on push to `main` when `web/src/locale/messages.xlf`, `content/blog/**`, or `content/wiki/pushup-types/**` changes. Manual "Run now" is always available from [claude.ai/code/routines](https://claude.ai/code/routines).
2. The routine re-extracts i18n (`pnpm nx run web:extract-i18n`), seeds fallbacks (`tools/src/sync-xliff-locales.mjs`), then runs `tools/src/detect-translation-gaps.mjs` to enumerate three gap classes:
   - XLIFF units in `messages.<lang>.xlf` with `state="initial"` AND `<target>` == `<source>`.
   - Missing `content/blog/<folder>/<lang>.md` files.
   - Missing `content/wiki/pushup-types/<id>.<lang>.md` files.
3. The detector emits a Markdown brief listing every gap with the source German text and the hard rules (preserve XLIFF placeholders, flip `state` to `translated`, quote YAML list items containing `: `, run `tools:generate-content` after content edits). The routine reads this brief and does the translation work itself in the same session — no Copilot dispatch hop.
4. The routine keeps **exactly one** open PR titled `chore(i18n): fill missing translations (<count> items)`, backed by the long-lived `claude/translations` branch. If that PR is already open it is **updated in place** (branch merged up to date with `main`, then re-pushed) instead of opening a second one. CI runs the normal lint/test/build pipeline on it. If there are no new gaps the routine opens nothing new — when no PR is open it just exits; when a PR is already open it refreshes that branch onto `main` (only if `main` advanced, so an unchanged state still costs nothing) to keep the PR mergeable, then leaves it to merge.

**When NOT to wait for the routine:**

- Fixing a bug in an existing translation (typo, broken `<ph>` placeholder, wrong meaning a reviewer flagged) — edit the target file directly in the same PR.
- Seeding a brand-new locale code — see "Adding a new locale" below; the routine only fills gaps in already-registered locales.
- Local dev where you need a translated string immediately — write it inline; the next routine run will leave it alone (it only touches `state="initial"` fallbacks).

**Manual dispatch:** [claude.ai/code/routines](https://claude.ai/code/routines) → "Translate XLIFF / content gaps" → "Run now". No `locales` / `dry_run` knobs any more — the routine always sweeps all locales and always exits cleanly when there are no gaps, so a manual run is safe at any time.

## XLIFF file management

- **Source XLIFF (`web/src/locale/messages.xlf`) is auto-generated — never edit it manually.** After changing `i18n`-marked templates, run `pnpm nx run web:extract-i18n` to regenerate it. Translation files (`messages.<lang>.xlf`) are written by the Claude translations routine; only edit them by hand to fix a specific bug in an existing translation.
- **XLF maintenance when moving text:** When moving i18n-annotated text between components, update the `<source>` in **both** XLF files to match the new template structure. Stale `<source>` with old placeholder names (`START_BLOCK_IF`, etc.) causes production build errors.

## `LOCALE_ID`

`LOCALE_ID` is set automatically by Angular i18n at build time — **NEVER provide it manually** in `app.config.ts`. Injecting it manually is fine: use `inject(LOCALE_ID)` to select locale-aware data at runtime.

## Date formatting

- Use locale-aware date pipe formats (`'longDate'`, `'short'`). Never hardcode a locale parameter like `'de'`.

## Number formatting

Raw `{{ value }}` of numerics in templates ignores `LOCALE_ID` — `5000` stays as `5000` regardless of locale. Pipe through `| number:'1.0-0'` so digits group per the active locale (`5,000` vs `5.000`):

```html
<b>{{ p.total | number: '1.0-0' }}</b>
```

The pipe consumes the active `LOCALE_ID` automatically; **do not** pass a hard-coded locale arg (third pipe arg) — that defeats the build-time locale split.

For strings built outside templates (share text, OG image content), use `value.toLocaleString('de-DE')` with the locale you want — those run on whatever runtime locale is active and don't go through Angular's pipe machinery.

## Mark brand strings with `i18n` even when the source already reads English

The source locale is `de`. Templates without an `i18n` attribute bypass XLIFF extraction entirely, so a hard-coded `Pushup Tracker` subtitle never gets a translation slot — even if the German and English text happen to be identical, marking it preserves the option for translators and keeps the XLIFF file consistent. Cost is one i18n id; benefit is no future "why is this string untranslatable" surprise.

## Dynamic / non-template content

Dynamic data (blog posts, feature descriptions, etc.) must be locale-aware too — XLIFF only covers templates and `$localize`. Use `inject(LOCALE_ID)` to pick the right data at runtime.

## Language switching

Implemented in `app.ts` `setLanguage()` via `window.location.replace()` with locale prefix (`/de/…`, `/en/…`). Must preserve:

- current path
- query params
- hash

Root paths need a trailing slash (`/de/`, `/en/`) to match Firebase hosting rewrites.

## Adding a new locale

Adding a new locale code (e.g. `no` for Norwegian) touches **7 places**. Miss one and either the build fails (`i18nMissingTranslation: error`) or the locale silently doesn't render. Checklist:

1. **`web/src/server-locale-redirect.ts`** — add to the `SUPPORTED_LOCALES` tuple. Drives `pickLocale`, the SSR redirect, the sitemap generator (regex-parsed), and `LocalePrefix` types throughout the app.
2. **`web/src/app/app.ts`** — add `{ code: '<lang>', label: '<self-name>' }` to `LANGUAGE_OPTIONS`. The label is the language's own name (e.g. `Norsk`, `Ελληνικά`) so a speaker recognises it regardless of the current UI locale.
3. **`web/project.json`** — add an entry under `i18n.locales` (translation file path + `subPath`) **and** add the code to all three `localize` arrays (production, production-preview, staging). Forgetting any of the three breaks that build configuration.
4. **`web/src/app/core/seo.service.ts`** — add `<lang>: '<ll_CC>'` to the `ogLocaleFor` map (e.g. `no: 'nb_NO'` — see [OpenGraph's accepted locales](https://developers.facebook.com/docs/opengraph/guides/internationalization/); for Norwegian use the Bokmål tag `nb_NO`, the macro `no_NO` is not in the list). TS won't compile without it because the map is `Record<LocalePrefix, string>`.
5. **`web/src/locale/messages.<lang>.xlf`** — required. Cheapest valid seed: `cp messages.en.xlf messages.<lang>.xlf` and change `trgLang="en"` to `trgLang="<lang>"`. Build passes because every `<target>` is present; translators can iteratively replace English text. Don't try to ship without this file — the production build hard-fails.
6. **Tests:** `web/src/app/app.spec.ts` and `web/src/server-locale-redirect.spec.ts` parametrise over the locale list (`it.each([...])`). Add the new code so the language-switch and Accept-Language tests cover it.
7. **`tools/src/fix-translated-yaml.mjs`** — add the code to `NON_SOURCE_LANGS` so the YAML fixer scans the new `<lang>.md` files.

The sitemap generator (`tools/src/generate-sitemap.js`) and the content generator (`tools/src/generate-content.mjs`) **do not** need updates — they auto-discover locales from `SUPPORTED_LOCALES` and from `<lang>.md` filenames respectively.

Translating actual blog/wiki content is a separate workstream — drop in `<lang>.md` files under `content/blog/<folder>/` and `content/wiki/pushup-types/`. Haiku is sufficient for these markdown translations; see "Machine-translated frontmatter pitfalls" below.

## Machine-translated frontmatter pitfalls

Translation agents writing `<lang>.md` frontmatter trip over YAML quoting in two distinct ways. Only the first is auto-fixed by `tools/src/fix-translated-yaml.mjs`:

1. **Apostrophes inside single-quoted scalars** — French/Italian "l'auteur", Norwegian compound words, etc. The fixer either doubles the apostrophe or switches the scalar to double quotes.
2. **Colon-space inside an unquoted list item** — _not_ fixed automatically. A line like `- Word: rest of sentence` parses as a YAML map (`Word` → `rest of sentence`), then `loadPushupTypeContent` rejects it: _"frontmatter field `instructions` item at index N must be a non-empty string (got object)"_. The fix is to wrap the whole item in single or double quotes:

   ```yaml
   instructions:
     - 'Skift til siden under nedstigningen: en arm bøyes, den andre forblir rett.'
   ```

   Run `node tools/src/generate-content.mjs` after every batch of translations — it surfaces this error immediately. When prompting a translation agent, explicitly call out the colon-space rule alongside the apostrophe rule; otherwise expect to manually fix a handful of items per language.

Prettier may rewrite the quote style (single ↔ double) on commit based on the scalar's content — e.g. it switches to double quotes when a single-quoted scalar contains an apostrophe. The agent's specific choice between `"..."` and `'...'` doesn't matter; what matters is that any colon-space item is quoted so YAML parses it as a string rather than a map.

## Root `/` handling

The root `/` is locale-aware out of the box. On Firebase App Hosting:

- `GET /` → `302 /de` (default) or `302 /en` (when `Accept-Language: en*`)
- Response includes `Vary: Accept-Language`

Angular SSR's locale middleware handles this — **do NOT reimplement it in `server.ts`**.
